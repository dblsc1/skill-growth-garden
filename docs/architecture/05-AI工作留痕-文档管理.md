# AI 工作留痕 & 文档管理

多 agent 自动化开发，最大的风险不是"做不出来",而是**半年后没人知道为什么这么做**。  
所以留痕规范照搬大公司正规项目的做法,核心原则:**给人看,可追溯,进 git。**

## 三套留痕,各司其职

| 制品 | 谁写 | 给谁看 | 类比业界 |
|---|---|---|---|
| **Git commit** | 仲裁者(唯一 push 者) | 看历史的人 | Conventional Commits |
| **devlog / ADR** | 各 worker / test 在交活前写 | 未来维护者 | Architecture Decision Records |
| **PIPELINE_LOG.md** | 仲裁者每步自动追加 | 项目经理 / 你 | CI/CD 审计日志 |

三者都 git 化。Agent 不直接 push(只有仲裁者能),但 agent **写文件**,仲裁者集成时一起提交。

## 1. Git Commit 规范(Conventional Commits)

仲裁者 push 时用结构化 commit message:

```
<type>(<scope>): <subject>

<body: 为什么这么做,不是做了什么>

Agents: worker=claude-code reviewer=codex test=codex
Task: <原始任务描述>
```

- **type**: `feat` / `fix` / `test` / `refactor` / `docs` / `chore` / `ci`
- **scope**: `backend` / `frontend` / `contracts` / `orchestrator`
- 例:`feat(backend): add /diary/extract endpoint`

contracts/ 变更必须单独成一个 commit,scope=`contracts`,方便审计 API 演进。

## 2. devlog —— 每个任务一篇(给人看的决策记录)

目录:`docs/devlog/YYYY-MM-DD-<short-slug>.md`  
谁写:干活的 worker 在写 `.agent_status.json` 之前,顺手写这篇。

```markdown
# 2026-06-19 — diary extract endpoint

## What
实现 POST /api/v1/diary/extract,调用 DeepSeek 提炼三层技能。

## Why
MVP 核心闭环的第一步。提取不写库(只有 /confirm 写),
所以本端点是纯函数式,方便测试和重试。

## Decisions (ADR-style)
- 用 DeepSeek 主、GPT-4o 兜底:成本/中文质量权衡
- 提取结果不落库:隐私(日记正文不上服务器)+ 幂等
- 返回结构含 note 字段:供用户在确认框编辑

## Contract impact
- 无 / 或:新增 ExtractResponse.note(additive,不破坏前端)

## Tests
- 单元:test_extract_three_tier(worker 自测)
- 待 E2E:确认框展示提取结果(交给前端 test)

## Agent
worker=claude-code, reviewer=codex
```

**关键:Why 和 Decisions 是重点。** 代码本身说明"做了什么",devlog 说明"为什么"。  
这是 ADR(Architecture Decision Record)的轻量版,大公司用来防止"知识随人离职蒸发"。

## 3. TESTLOG —— 测试工程师的留痕

`tests/backend/TESTLOG.md` / `tests/frontend/TESTLOG.md`,每轮测试追加一段。  
见 backend-test / frontend-test 的 CLAUDE.md。重点记:加了哪些用例、结果、契约校验结论。

## 4. PIPELINE_LOG.md —— 仲裁者的调度审计

`docs/PIPELINE_LOG.md`,仲裁者每个决策自动追加一行(graph.py 的 `append_pipeline_log`):

```markdown
- `2026-06-19 14:02:11` TASK: implement /diary/extract endpoint
- `2026-06-19 14:02:13` PLAN: backend-only, complex → claude-code solo first
- `2026-06-19 14:18:40` [build:backend:solo] 1 report collected
- `2026-06-19 14:19:02` [dispatch] test: want API integration coverage
- `2026-06-19 14:31:55` [test:backend] PASS: 12 API tests, contract OK
- `2026-06-19 14:32:10` [dispatch] integrate: backend good, no frontend change
- `2026-06-19 14:33:20` [integration] clean
- `2026-06-19 14:33:25` [push] branch=feature/diary-extract
```

这是流程的黑匣子。出问题时,这里能还原"仲裁者当时怎么想、派了谁、谁通过了"。

## 仓库里的文档结构(实际 repo)

```
growth-garden/
├── docs/
│   ├── PIPELINE_LOG.md          # 仲裁者自动维护
│   ├── devlog/                  # 每任务一篇,worker 写
│   │   └── 2026-06-19-diary-extract.md
│   └── adr/                     # 重大架构决策(可选,从 devlog 提炼)
│       └── 0001-no-diary-text-on-server.md
├── tests/
│   ├── backend/TESTLOG.md
│   └── frontend/TESTLOG.md
└── CHANGELOG.md                 # 面向用户的版本变更(发版时整理)
```

## 谁在 CLAUDE.md / AGENTS.md 里被要求留痕

| Agent | 留痕义务 |
|---|---|
| backend worker | 交活前写 `docs/devlog/` 一篇 + commit 由仲裁者代提 |
| frontend worker | 同上 |
| backend test | 写 `tests/backend/TESTLOG.md` |
| frontend test | 写 `tests/frontend/TESTLOG.md` |
| 仲裁者 | 维护 `docs/PIPELINE_LOG.md` + 写规范 commit + push |

## 为什么这么设计

- **只有仲裁者 push**:单一提交入口,commit 质量可控,避免 N 个 agent 乱 push 脏历史
- **agent 写文件、仲裁者提交**:留痕和代码在同一个 commit 里,天然对应
- **devlog 强制写 Why**:对抗 AI 代码"能跑但没人懂"的最大隐患
- **PIPELINE_LOG 自动化**:调度决策不靠人记,出事能复盘
